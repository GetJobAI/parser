# Пояснення до діаграми `use_case_resume_parser.puml`

## Що показує ця діаграма

Ця use case-діаграма показує `Resume Parser Service` **з точки зору зовнішньої взаємодії**, а не внутрішньої реалізації.

Тобто вона відповідає на три прості питання:

- хто взаємодіє із сервісом;
- з якою метою;
- які зовнішні сценарії сервіс підтримує.

Ця діаграма спеціально побудована **саме для твого модуля**, а не для всієї системи загалом.

---

## Чому ця діаграма тепер правильніша

Після зауважень викладача use case потрібно було спростити й зробити більш коректним.

У правильній use case-діаграмі не варто показувати:

- внутрішні кроки пайплайна;
- технічні деталі на кшталт валідації MIME type;
- створення запису в БД;
- fallback, quality checker, section parsing як окремі use case.

Це не зовнішні сценарії використання, а **внутрішня логіка сервісу**.

Тому в цій версії залишені лише ті сценарії, які справді видно ззовні.

---

## Як читати діаграму

У центрі діаграми знаходиться прямокутник `Resume Parser Service`.

Це межа системи. Усе, що всередині, належить до сервісу. Усе, що зовні, є зовнішніми учасниками взаємодії.

На діаграмі є:

- **actors** — зовнішні учасники;
- **use cases** — зовнішні сценарії використання сервісу.

---

## Актори

### `API Gateway`
**Українська назва:** `API Gateway`

Це зовнішній системний актор, через який до сервісу надходить основний робочий запит.

Для твого модуля це логічний основний актор, тому що parser service не працює напряму з користувачем, а отримує запит через gateway.

Саме gateway:

- передає файл;
- передає trusted user context;
- запускає основний сценарій парсингу.

### `Frontend Developer`
**Українська назва:** `Фронтенд-розробник`

Цей актор не запускає основний бізнес-сценарій парсингу напряму, але взаємодіє із сервісом для інтеграції.

Його цікавлять допоміжні сценарії:

- отримання контракту структури даних;
- перегляд API-документації.

Це важливо, бо твій сервіс виконує не лише бізнес-функцію парсингу, а ще й надає інтеграційний інтерфейс для інших розробників.

---

## Use cases

### `Parse uploaded resume`
**Українська назва:** `Розпарсити завантажене резюме`

Це головний use case усього сервісу.

Він означає, що зовнішня система передає в parser service файл резюме, а сервіс:

- приймає документ;
- обробляє його;
- перетворює в структуровані дані;
- повертає результат парсингу.

На use case-діаграмі цей сценарій показаний як один цілісний зовнішній прецедент.

Це правильно, бо з точки зору зовнішнього актора важливо не те, як саме сервіс усередині:

- виконує extraction;
- визначає секції;
- запускає fallback;
- перевіряє якість;

а те, що він **виконує парсинг резюме як завершену зовнішню послугу**.

### `Get content contract`
**Українська назва:** `Отримати контракт структури content`

Цей use case потрібен для інтеграції frontend або інших клієнтів із сервісом.

Він означає, що зовнішній розробник може отримати:

- опис структури `content`;
- очікувану форму JSON;
- допоміжні приклади полів.

Це особливо важливо для узгодження структури між parser service, frontend і template service.

### `View API documentation`
**Українська назва:** `Переглянути API-документацію`

Цей use case означає, що зовнішній розробник може відкрити документацію сервісу.

У практичному сенсі це доступ до Swagger / OpenAPI опису:

- endpoint-ів;
- моделей відповіді;
- базових сценаріїв використання.

Це допоміжний, але реальний зовнішній сценарій взаємодії з сервісом.

---

## Послідовне пояснення всієї діаграми

Якщо пройтися по діаграмі зліва направо, логіка така:

1. `API Gateway` звертається до use case `Parse uploaded resume`.
2. Це означає, що gateway передає резюме у твій сервіс для обробки.
3. Окремо `Frontend Developer` може звернутися до `Get content contract`.
4. Це потрібно, щоб зрозуміти, яку саме JSON-структуру повертає або зберігає сервіс.
5. Також `Frontend Developer` може звернутися до `View API documentation`.
6. Це потрібно для зручної інтеграції через документацію endpoint-ів.

Тобто на діаграмі показані:

- один основний бізнес-сценарій;
- два допоміжні інтеграційні сценарії.

---

## Що свідомо не включено в цю діаграму

У цю use case-діаграму спеціально **не включені**:

- PostgreSQL;
- pipeline steps;
- section detector;
- quality checker;
- fallback manager;
- внутрішні parser-и;
- створення запису в БД;
- повернення `resume_id` як окремий use case.

Причина проста: це не зовнішні сценарії використання, а внутрішні технічні механізми реалізації сервісу.

Такі речі краще показувати на:

- activity-діаграмах;
- component-діаграмах;
- sequence-діаграмах.

---

## Як пояснювати цю діаграму на захисті

Можна казати так:

> Ця use case-діаграма показує Resume Parser Service з точки зору зовнішніх сценаріїв використання. Основний сценарій — це парсинг завантаженого резюме, який запускається через API Gateway. Додатково сервіс підтримує два інтеграційні сценарії для розробників: отримання контракту структури `content` і перегляд API-документації. Внутрішні кроки парсингу тут не показуються, тому що для них призначені activity- та component-діаграми.
