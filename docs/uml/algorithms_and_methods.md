# Алгоритми та методи, використані в сервісі парсингу

## Для чого потрібен цей розділ

У підрозділі про алгоритми та методи потрібно показати не тільки архітектуру сервісу, а й логіку його роботи. Для цього доцільно описати, які саме етапи проходить документ під час обробки, які правила використовуються на кожному етапі та якими діаграмами це підтверджується.

У даному проєкті під алгоритмами слід розуміти не класичні математичні алгоритми, а послідовності дій, за допомогою яких сервіс:

- приймає документ;
- витягає з нього текст;
- відновлює логічну структуру;
- розподіляє дані по полях;
- перевіряє якість результату;
- зберігає підсумковий структурований вміст.

## Основні алгоритми, через які проходить парсинг

Нижче наведено основні алгоритми, що використовуються в поточній реалізації сервісу.

## 1. Алгоритм валідації вхідного запиту

### Призначення

Перед запуском парсингу сервіс має переконатися, що запит коректний і його можна обробляти далі.

### Де реалізований

- [app/api/resumes.py](/home/kerio/Documents/nure/s_6/kkp/parcer/app/api/resumes.py)

### Основні перевірки

- наявність trusted header від gateway;
- наявність файлу;
- допустимість розширення;
- допустимість MIME-типу;
- перевірка, що файл не порожній.

### Результат

Якщо запит некоректний, сервіс одразу завершує обробку і повертає помилку валідації. Якщо все коректно, формується початковий об’єкт результату та створюється запис обробки.

## 2. Алгоритм вибору шляху обробки за типом документа

### Призначення

Сервіс підтримує два типи документів:

```
PDF
DOCX
```

Тому перед початком витягання тексту треба визначити, який шлях обробки використати.

### Де реалізований

- [app/pipeline/resume_pipeline.py](/home/kerio/Documents/nure/s_6/kkp/parcer/app/pipeline/resume_pipeline.py)

### Суть алгоритму

1. Аналізується ім’я файлу або його розширення.
2. Якщо документ є PDF, викликається PDF-extractor.
3. Якщо документ є DOCX, викликається DOCX-extractor.
4. В іншому випадку документ відхиляється як непідтримуваний.

## 3. Алгоритм витягання тексту з документа

### Призначення

Мета цього алгоритму — перетворити двійковий файл у внутрішнє текстове представлення, придатне для подальшого аналізу.

### Де реалізований

- [app/extractors/pdf_extractor.py](/home/kerio/Documents/nure/s_6/kkp/parcer/app/extractors/pdf_extractor.py)
- [app/extractors/docx_extractor.py](/home/kerio/Documents/nure/s_6/kkp/parcer/app/extractors/docx_extractor.py)

### Суть алгоритму

Для PDF і DOCX використовуються різні механізми extraction, але обидва вони зводяться до одного спільного внутрішнього формату:

```
TextBlock
```

Кожен текстовий блок містить:

- текст;
- порядок у документі;
- для PDF також координати розташування.

### Додаткові результати

На цьому етапі також збираються технічні ознаки документа:

- наявність графіки;
- наявність нестандартних шрифтів;
- ознаки складного layout;
- ознаки header/footer фрагментів.

## 4. Алгоритм попередньої обробки текстових блоків

### Призначення

Після extraction текст ще не готовий до прямого структурного розбору. Він може містити:

- зайві пробіли;
- злиті рядки;
- пошкоджені маркери списків;
- повторювані header/footer блоки;
- змішані фрагменти кількох секцій.

### Де реалізований

- [app/preprocess/cleaner.py](/home/kerio/Documents/nure/s_6/kkp/parcer/app/preprocess/cleaner.py)

### Основні кроки

1. Нормалізація пробілів.
2. Нормалізація bullet-префіксів.
3. Видалення порожніх і дубльованих блоків.
4. Виявлення повторюваних header/footer фрагментів.
5. Обережне об’єднання або розбиття рядків, якщо структура документа цього потребує.

### Метод

Тут використовується набір евристичних правил очищення тексту. Його мета — не змінити зміст, а лише зробити документ придатним до подальшого аналізу.

## 5. Алгоритм визначення layout документа

### Призначення

### Основна ідея

## Текст, який можна використати в пояснювальній записці

У сервісі парсингу резюме використовується поетапна алгоритмічна обробка документа. Після валідації вхідного запиту система обирає шлях обробки за типом документа, виконує витягання тексту, попередню очистку блоків, визначення layout та виявлення секцій. Далі для кожної секції застосовуються спеціалізовані алгоритми розбору, зокрема алгоритм парсингу контактних даних, освіти, досвіду роботи та спискових секцій. Для секції досвіду додатково використовується евристичний алгоритм аналізу заголовка запису, який дозволяє виділяти назву компанії, посаду, локацію та дати. Якщо основний шлях парсингу дає неповний результат, запускається fallback-обробка. Після цього застосовується правило-орієнтований алгоритм оцінки якості результату та фіксується фінальний стан обробки документа.

## Висновок

У даному проєкті під алгоритмами і методами доцільно розуміти:

- алгоритми валідації запиту;
- алгоритми extraction тексту;
- алгоритми попередньої обробки;
- алгоритми визначення структури документа;
- алгоритми секційного парсингу;
- fallback-алгоритми;
- алгоритми контролю якості;
- алгоритм керування станами обробки.

Таким чином, підрозділ про алгоритми можна змістовно заповнити не лише двома activity-діаграмами, а описом усього маршруту, який проходить документ від моменту отримання до моменту формування структурованого результату.
