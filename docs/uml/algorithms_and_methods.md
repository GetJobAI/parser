## 1. Алгоритм загального парсингу резюме

### Призначення

Цей алгоритм є центральним для всього сервісу. Він описує повний шлях обробки резюме:

- від моменту отримання файлу;
- до моменту збереження структурованого JSON у базу даних.

### Де реалізований

- [app/pipeline/resume_pipeline.py](/home/kerio/Documents/nure/s_6/kkp/parcer/app/pipeline/resume_pipeline.py)

### Основні кроки

1. Створення початкового об’єкта `ResumeContent` зі статусом `processing`.
2. Вибір шляху extraction залежно від типу файлу:
   - `PDFExtractor` для PDF;
   - `DOCXExtractor` для DOCX.
3. Отримання текстових блоків у спільному внутрішньому форматі `TextBlock`.
4. Попередня очистка блоків:
   - нормалізація пробілів;
   - нормалізація bullet-ів;
   - обережне злиття або розбиття рядків;
   - видалення повторюваних header/footer-фрагментів.
5. Аналіз layout документа і відновлення правильного порядку читання.
6. Пошук заголовків секцій.
7. Побудова меж секцій.
8. Окремий парсинг кожної секції.
9. Застосування fallback-логіки.
10. Формування `unassigned_blocks`.
11. Оцінка якості результату.
12. Встановлення фінальних метаданих і повернення результату.

### Яка діаграма це показує

- [activity_parse_pipeline.puml](/home/kerio/Documents/nure/s_6/kkp/parcer/docs/uml/activity_parse_pipeline.puml)

### Чому це алгоритм

Це алгоритм, бо сервіс виконує **чітко визначену послідовність кроків**, де результат кожного наступного етапу залежить від попереднього.

## 2. Алгоритм extraction тексту з документа

### Призначення

Мета цього алгоритму — перетворити вхідний файл у внутрішнє представлення, придатне для подальшого аналізу.

### Де реалізований

- [app/extractors/pdf_extractor.py](/home/kerio/Documents/nure/s_6/kkp/parcer/app/extractors/pdf_extractor.py)
- [app/extractors/docx_extractor.py](/home/kerio/Documents/nure/s_6/kkp/parcer/app/extractors/docx_extractor.py)

### Суть методу

У сервісі використано **два різні методи extraction**:

- для PDF — extraction через `PyMuPDF`;
- для DOCX — extraction через `python-docx`.

Після цього обидва результати зводяться до єдиного формату `TextBlock`, щоб решта pipeline не залежала від типу вхідного документа.

### Чому це важливо

Це дозволяє:

- уникнути конвертації DOCX у PDF;
- обробляти різні типи документів окремо;
- використовувати один спільний pipeline далі.

### Який метод тут застосований

Тут використано метод **уніфікації вхідних даних**: різні джерела даних переводяться в спільне проміжне представлення.

## 3. Алгоритм попередньої обробки текстових блоків

### Призначення

Після extraction текст зазвичай ще непридатний до прямого парсингу. Він містить:

- зайві пробіли;
- злиті рядки;
- некоректні bullet-и;
- повторювані header/footer-фрагменти;
- змішані секції в одному блоці.

### Де реалізований

- [app/preprocess/cleaner.py](/home/kerio/Documents/nure/s_6/kkp/parcer/app/preprocess/cleaner.py)

### Основні прийоми

- нормалізація пробілів;
- нормалізація префіксів списків;
- видалення пустих і дубльованих блоків;
- обережне об’єднання логічно пов’язаних рядків;
- розбиття одного блоку на кілька, якщо в ньому злиті:
  - section headers;
  - bullets;
  - початок наступної позиції досвіду.

### Який метод тут застосований

Тут використовується **евристична очистка тексту**, тобто набір практичних правил, що покращують структуру тексту без втрати змісту.

### Чому це важливо

Без цього етапу секції резюме можуть визначатися неправильно, а частина даних буде або загублена, або прив’язана не до тієї секції.

## 4. Алгоритм визначення layout документа

### Призначення

PDF-документи можуть мати:

- одноколонкову структуру;
- двоколонкову структуру;
- змішану структуру.

Щоб коректно відновити порядок читання, сервіс аналізує координати блоків.

### Де реалізований

- [app/preprocess/layout.py](/home/kerio/Documents/nure/s_6/kkp/parcer/app/preprocess/layout.py)

### Суть методу

Для кожного блоку враховуються його координати:

- `x0`, `y0`, `x1`, `y1`

На основі розташування блоків сервіс намагається визначити:

- чи документ схожий на `single_column`;
- чи документ схожий на `two_column`.

Після цього порядок блоків може бути перебудований.

### Який метод тут застосований

Тут використовується **просторовий евристичний аналіз layout**.

## 5. Алгоритм виявлення секцій резюме

### Призначення

Потрібно визначити, де в документі знаходяться секції:

- contact;
- summary;
- experience;
- education;
- skills;
- certifications;
- languages;
- projects.

### Де реалізований

- [app/detectors/section_detector.py](/home/kerio/Documents/nure/s_6/kkp/parcer/app/detectors/section_detector.py)
- [app/detectors/boundaries.py](/home/kerio/Documents/nure/s_6/kkp/parcer/app/detectors/boundaries.py)

### Основні кроки

1. Вибір коротких текстових блоків, схожих на заголовки.
2. Нормалізація тексту.
3. Порівняння з відомими назвами секцій і синонімами.
4. Використання fuzzy matching для випадків:
   - typo;
   - незначних відхилень;
   - альтернативних назв.
5. Визначення canonical section name.
6. Побудова меж секції від одного заголовка до наступного.

### Який метод тут застосований

Тут поєднано два методи:

- **словниковий метод**;
- **fuzzy matching**.

### Чому це важливо

Реальні CV часто мають нестандартні заголовки, наприклад:

- `Work History`
- `Professional Experience`
- `Ekucation`
- `Skils`

Точний пошук по одному слову тут був би недостатнім.

## 6. Алгоритм парсингу досвіду роботи

### Призначення

Секція `experience` є однією з найскладніших, бо в ній треба:

- визначити окремі позиції роботи;
- знайти назву компанії;
- знайти посаду;
- витягнути дати;
- виділити bullets;
- зберегти сирий текст, якщо структура неповна.

### Де реалізований

- [app/parsers/experience_parser.py](/home/kerio/Documents/nure/s_6/kkp/parcer/app/parsers/experience_parser.py)

### Основні кроки

1. Об’єднання тексту секції.
2. Розбиття секції на окремі записи досвіду.
3. Пошук header-частини кожного запису.
4. Виявлення діапазону дат через регулярні вирази.
5. Евристичний поділ header-а на:
   - `company`
   - `title`
   - `location`
6. Витяг bullet-пунктів.
7. Збереження `description_raw` або `raw_text`, якщо повного структурного розбору зробити не вдалося.

### Який метод тут застосований

Тут використовується **правило-орієнтований евристичний парсинг**.

### Чому це важливо

Формат досвіду роботи дуже відрізняється від документа до документа, тому жорсткий шаблон тут не працює.

## 8. Алгоритм парсингу спискових секцій

### Призначення

До таких секцій належать:

- skills;
- certifications;
- languages;
- projects.

### Де реалізований

- [app/parsers/generic_list_parser.py](/home/kerio/Documents/nure/s_6/kkp/parcer/app/parsers/generic_list_parser.py)
- [app/parsers/skills_parser.py](/home/kerio/Documents/nure/s_6/kkp/parcer/app/parsers/skills_parser.py)
- [app/parsers/languages_parser.py](/home/kerio/Documents/nure/s_6/kkp/parcer/app/parsers/languages_parser.py)
- [app/parsers/certifications_parser.py](/home/kerio/Documents/nure/s_6/kkp/parcer/app/parsers/certifications_parser.py)
- [app/parsers/projects_parser.py](/home/kerio/Documents/nure/s_6/kkp/parcer/app/parsers/projects_parser.py)

### Суть підходу

Спочатку використовується загальний list-like parsing:

- розбиття за комами;
- розбиття за `•`;
- розбиття за `;`;
- розбиття за `|`;
- розбиття за `·` у тих секціях, де це доречно.

Потім для окремих секцій використовуються додаткові правила.

Наприклад:

- для `skills` враховуються категорії;
- для `languages` виділяються окремі мовні елементи;
- для `projects` використовуються межі між окремими проєктами;
- для `certifications` — межі між окремими сертифікаціями.

### Який метод тут застосований

Тут використовується **комбінований метод**:

- базовий generic parser;
- спеціалізовані евристики для конкретних секцій.

## Які алгоритми доцільно згадувати в поясненні до діаграм

Якщо тобі треба короткий список для документації або захисту, то достатньо згадувати ці 5:

1. Загальний алгоритм парсингу резюме.
2. Алгоритм виявлення секцій через словник і fuzzy matching.
3. Алгоритм евристичного парсингу досвіду роботи.
4. Алгоритм fallback-обробки слабких результатів.
5. Алгоритм rule-based оцінки якості.

Саме вони найкраще показують, що в сервісі є **реальна алгоритмічна логіка**, а не просто набір CRUD-операцій.

## Що можна прямо написати в пояснювальній записці

У даному ККП використовуються алгоритми поетапної обробки текстових документів. Основним є алгоритм загального парсингу резюме, який включає extraction тексту, попередню обробку блоків, визначення layout, пошук заголовків секцій, побудову меж секцій, спеціалізований парсинг кожної секції, fallback-обробку та оцінку якості результату. Для визначення секцій використовується словниково-евристичний підхід із fuzzy matching, що дозволяє враховувати синоніми та незначні помилки в заголовках. Для секцій `experience` та `education` застосовується правило-орієнтований евристичний парсинг. Для контролю повноти результату використовується алгоритм rule-based quality checking. Життєвий цикл обробки резюме подано у вигляді діаграми станів, а послідовність основного process flow — у вигляді діаграми активностей.
