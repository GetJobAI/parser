## Загальні терміни

### `service` / сервіс

Окрема програма або модуль, який виконує одну конкретну задачу.  
У цьому проєкті сервіс приймає резюме, аналізує його і зберігає результат.

### `microservice` / мікросервіс

Невеликий окремий сервіс у складі великої системи.  
Замість однієї великої програми система ділиться на кілька маленьких сервісів.

### `gateway`

Проміжний сервіс перед основним API.  
Зазвичай він:

- приймає зовнішні запити;
- перевіряє автентифікацію;
- перенаправляє запит у потрібний внутрішній сервіс.

### `HTTP`

Протокол, через який клієнт і сервер обмінюються запитами та відповідями в інтернеті або локальній мережі.

### `API`

Інтерфейс для взаємодії між програмами.  
У цьому проєкті API означає endpoint, на який можна надіслати резюме.

### `endpoint`

Конкретна адреса API, наприклад:

```text
POST /resumes/parse
```

Тобто це "точка входу" в сервіс.

### `request`

Запит, який надсилає клієнт на сервер.

### `response`

Відповідь сервера на запит клієнта.

### `header`

Службова частина HTTP-запиту або відповіді.  
У цьому проєкті через header передається `user_id`.

### `trusted header`

Заголовок, якому сервіс довіряє, бо він був доданий іншим безпечним компонентом системи, наприклад gateway.

### `multipart/form-data`

Формат HTTP-запиту для надсилання файлів та форм.

## Архітектурні терміни

### `architecture` / архітектура

Загальна будова системи:

- з яких частин вона складається;
- як ці частини взаємодіють;
- хто за що відповідає.

### `layer` / шар

Рівень логіки в архітектурі.  
Наприклад:

- API layer
- DB layer
- parsing layer

### `module` / модуль

Окремий логічний шматок системи.  
Зазвичай це файл або група файлів з однією відповідальністю.

### `component`

Окрема частина системи, більша або загальніша за функцію.  
Наприклад parser, extractor, repository.

### `pipeline`

Послідовний ланцюг етапів обробки даних.  
У цьому проєкті pipeline означає:

1. extraction
2. preprocess
3. layout analysis
4. section detection
5. parsing
6. fallback
7. quality check

### `orchestrator`

Компонент, який не виконує всю роботу сам, а керує порядком запуску інших частин системи.  
У цьому проєкті `ResumePipeline` виступає як orchestrator.

### `runtime`

Період, коли програма реально виконується.

### `flow`

Послідовність проходження даних або викликів у системі.

### `lifecycle`

Життєвий цикл об’єкта або сервісу: створення, використання, завершення.

## Терміни обробки даних

### `parse` / парсити

Розбирати текст або дані на структуровані частини.

Наприклад:

- було: звичайний текст резюме;
- стало: окремо `name`, `email`, `experience`, `skills`.

### `parser`

Модуль, який виконує парсинг.

### `resume parsing`

Процес автоматичного аналізу резюме і перетворення його в структурований формат.

### `rules-first`

Підхід, у якому система спирається в першу чергу на правила, регулярні вирази, словники і евристики, а не на AI.

### `deterministic`

Передбачуваний.  
Тобто при однаковому вході система дає однаковий результат.

### `heuristic` / евристика

Практичне наближене правило, яке часто працює добре, але не гарантує ідеальність у всіх випадках.

Приклад:

- якщо в рядку є дата, назва компанії і bullet-и, то це, ймовірно, experience.

### `fallback`

Запасний шлях дії, який використовується, коли основний шлях не дав хорошого результату.

### `OCR`

Технологія розпізнавання тексту з картинки або скану.  
Потрібна для PDF, де немає текстового шару.

### `AI`

Штучний інтелект.  
У цьому проєкті він поки що не використовується як обов’язкова частина MVP.

### `MVP`

Minimal Viable Product — мінімально працездатна версія продукту.  
Тобто не ідеальна фінальна система, а перша корисна реалізація.

## Терміни структури документа

### `TextBlock`

Внутрішня модель одного шматка тексту.  
Це може бути:

- абзац;
- рядок;
- блок тексту, витягнутий з PDF.

### `block`

Один фрагмент тексту в документі.

### `layout`

Візуальна структура документа на сторінці:

- один стовпець;
- два стовпці;
- розташування тексту по координатах.

### `single-column`

Макет документа в один стовпець.

### `two-column`

Макет документа у два стовпці.

### `reading order`

Порядок, у якому документ треба читати, щоб сенс був правильний.

### `section`

Логічний розділ резюме:

- summary
- experience
- education
- skills

### `section header`

Заголовок секції.  
Наприклад `EXPERIENCE` або `EDUCATION`.

### `section boundary`

Межа секції: де секція починається і де закінчується.

### `header area`

Верхня частина документа, де зазвичай містяться:

- ім’я;
- контакти;
- інколи location.

### `unassigned_blocks`

Блоки тексту, які система не змогла впевнено віднести до жодної секції.

### `raw_text`

Сирий, неструктурований текст секції або запису.  
Використовується, коли сервіс не впевнений у точному розборі.

## Терміни форматів файлів

### `PDF`

Формат документа, який орієнтований на відображення сторінки.  
Його важче парсити, бо текст часто "живе" як графічний layout.

### `DOCX`

Формат документа Microsoft Word.  
Зазвичай він краще структурований для читання програмою.

### `MIME type`

Тип файлу, який передається в HTTP-запиті.  
Наприклад:

- `application/pdf`
- `application/vnd.openxmlformats-officedocument.wordprocessingml.document`

## Технічні терміни коду

### `schema`

Опис структури даних.  
У цьому проєкті schema — це Pydantic-модель.

### `contract`

Домовлений формат даних між частинами системи.

Наприклад `ResumeContent` — це контракт фінального JSON.

### `model`

Структура даних у коді.  
У цьому проєкті це переважно моделі Pydantic.

### `repository`

Компонент, який відповідає за доступ до БД і приховує SQL-логіку від решти системи.

### `client`

Компонент для підключення до зовнішнього ресурсу, наприклад до PostgreSQL.

### `pool`

Набір готових з’єднань до БД, які повторно використовуються, щоб не відкривати нове з’єднання щоразу.

### `dependency`

Залежність.  
Може означати:

- бібліотеку, яку використовує проєкт;
- або об’єкт, який потрібен іншому об’єкту для роботи.

### `dependency injection`

Підхід, коли потрібні залежності передаються об’єкту ззовні, а не створюються жорстко всередині.

### `helper`

Допоміжна функція або модуль.

### `utility`

Невеликий технічний інструмент або допоміжний код загального призначення.

### `wrapper`

Обгортка над якоюсь бібліотекою або функцією, щоб використання було простішим і контрольованішим.

### `thin wrapper`

Дуже легка обгортка, яка додає мало логіки.

### `hook`

Спеціально залишене місце в архітектурі, куди зручно додати нову логіку в майбутньому.

### `extensible`

Такий, який можна розширювати без повного переписування системи.

### `production-style`

Код, написаний у стилі, близькому до реальних промислових сервісів:

- модульний;
- читабельний;
- із розділенням відповідальностей;
- з обробкою помилок.

## Термінологія алгоритмів і пошуку

### `regex`

Скорочення від regular expression — регулярний вираз.  
Це шаблон для пошуку тексту.

### `fuzzy matching`

Нечітке порівняння рядків.  
Дозволяє сказати, що `Ekucation` і `Education` дуже схожі, навіть якщо це не точний збіг.

### `RapidFuzz`

Бібліотека Python для нечіткого порівняння тексту.

### `score`

Числова оцінка схожості або якості збігу.

### `candidate`

Кандидат, тобто варіант, який система розглядає як можливий.

Наприклад:

- candidate header;
- candidate section.

### `synonym`

Синонім.  
У section detection це різні назви однієї логічної секції.

### `pattern`

Шаблон або повторювана текстова форма.

### `classifier`

Модуль або логіка, що відносить об’єкт до певного класу.  
У широкому сенсі fallback і section heuristics теж виступають як прості класифікатори.

## Терміни якості та помилок

### `warning`

Попередження.  
Не критична помилка, але сигнал, що результат може бути неповним або слабким.

### `partial_parse`

Ознака того, що резюме розібране лише частково.

### `parse_status`

Поточний стан обробки:

- `processing`
- `completed`
- `failed`

### `parse_error`

Текст помилки, якщо обробка завершилась невдачею.

### `quality checker`

Модуль перевірки якості парсингу.

### `major sections`

Основні розділи резюме, наприклад:

- contact
- summary
- experience
- education
- skills

## Терміни тестування і розробки

### `CLI`

Command Line Interface — запуск програми через командний рядок.

### `local tooling`

Локальні інструменти для розробки та перевірки, які не є частиною основного API.

### `test case`

Окремий сценарій перевірки коду.

### `unit test`

Тест, який перевіряє одну невелику частину системи, наприклад одну функцію або один модуль.

### `debug`

Пошук і аналіз помилки або неправильної поведінки.

### `artifact`

Файл або результат, який був створений під час роботи програми.  
Наприклад `.parsed.md` після локального запуску.

## Терміни бази даних

### `PostgreSQL`

Реляційна база даних, у яку сервіс зберігає результат.

### `JSONB`

Спеціальний тип у PostgreSQL для зберігання JSON у зручному для запитів вигляді.

### `row`

Один запис у таблиці бази даних.

### `table`

Таблиця в базі даних.

### `UUID`

Унікальний ідентифікатор великої довжини, який майже гарантовано не повторюється.

## Терміни Python і FastAPI

### `FastAPI`

Python-фреймворк для створення HTTP API.

### `Pydantic`

Бібліотека для опису і валідації структурованих даних через моделі.

### `async`

Асинхронний режим роботи, коли програма може не блокувати виконання на очікуванні вводу-виводу, наприклад БД чи мережі.

### `lifespan`

Механізм у FastAPI для коду, який виконується при старті й завершенні застосунку.

### `state`

Спільне місце всередині FastAPI app, де можна зберігати об’єкти, наприклад repository або pipeline.
